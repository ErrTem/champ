## Trust Boundaries
| Boundary | Description |
|----------|-------------|
| Stripe → API | Public webhook endpoint; must authenticate requests via signature verification on raw body. |
| client → API (polling) | Authenticated user requests booking detail; must be scoped to owner. |
| API → DB | Must apply confirmation + slot consumption atomically and idempotently. |

## STRIDE Threat Register
| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08 | Spoofing | `POST /stripe/webhook` | mitigate | Verify `Stripe-Signature` using Stripe SDK `constructEvent` with raw body; reject invalid signatures. |
| T-04-09 | Replay | webhook event delivery | mitigate | Persist `event.id` with unique constraint; on duplicate return 200 without re-applying (D-09). |
| T-04-10 | Tampering | booking/slot transition | mitigate | Use Prisma `$transaction` to update booking + slot together; guarded slot update prevents double-consumption (D-10). |
| T-04-11 | Information Disclosure | webhook payload + logs | mitigate | Do not log payloads; store only `event.id` and minimal fields; metadata only `bookingId` (D-05). |
| T-04-12 | Information Disclosure | `GET /bookings/:id` | mitigate | User-scoped auth guard; ensure response excludes user PII beyond existing ids/status/time. |
</threat_model>
