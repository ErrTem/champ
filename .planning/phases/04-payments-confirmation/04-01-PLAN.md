<threat_model>
## Trust Boundaries
| Boundary | Description |
|----------|-------------|
| client → API | Untrusted client initiates checkout; must not influence price (D-06). |
| Stripe → API (webhook) | Public endpoint; must be gated by signature verification on raw body. |
| API → DB | Atomic state transitions required to prevent partial confirmation/slot consumption. |

## STRIDE Threat Register
| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | `POST /stripe/webhook` | mitigate | Add automated test asserting invalid/missing signature is rejected (401/400) and code uses Stripe SDK verification on raw body. |
| T-04-02 | Tampering | Checkout session amount | mitigate | Add automated test asserting unit_amount is derived from `Service.priceCents` in DB, not from client input (per D-06). |
| T-04-03 | Replay | Webhook retries/duplicates | mitigate | Add automated test asserting duplicate `event.id` is ignored (per D-09). |
| T-04-04 | Information Disclosure | Logging of webhook payloads | mitigate | Ensure tests (and later implementation) never log full payload; metadata contains only `bookingId` (per D-05). |
</threat_model>
