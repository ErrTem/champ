<threat_model>
## Trust Boundaries
| Boundary | Description |
|---------|-------------|
| Client → Admin API | Cookie-authenticated requests must be authorized as admin (do not trust client UI gating). |
| Admin UI → Router | Route guards improve UX but must not be only enforcement. |
| Seed/env → DB | Admin provisioning uses secrets; must avoid logging plaintext password. |

## STRIDE Threat Register
| Threat ID | Category | Component | Disposition | Mitigation Plan |
|----------|----------|-----------|-------------|-----------------|
| T-06-01 | E (Elevation) | `/admin/*` controllers | mitigate | Enforce `@UseGuards(JwtAccessAuthGuard, AdminGuard)` server-side; `AdminGuard` DB-checks `User.isAdmin` per request. |
| T-06-02 | S (Spoofing) | Admin provisioning via env | mitigate | Use bcrypt hashing; never log `ADMIN_PASSWORD`; upsert by email; require both env vars present to create admin. |
| T-06-03 | I (Information disclosure) | `/users/me` response | mitigate | Return safe user DTO only; include `isAdmin` but never include passwordHash/refresh secrets. |
| T-06-04 | T (Tampering) | Frontend-only gating | mitigate | Keep frontend `adminGuard` + hidden Profile link for UX, but treat as non-security; backend guard is authoritative. |
</threat_model>
