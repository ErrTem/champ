Create a new Nest module `AvailabilityModule` exposing a public endpoint:

`GET /availability`

Query params (validated DTO):
- `fighterId: string` (required)
- `serviceId: string` (required)
- Optional:
  - `fromDate` as `YYYY-MM-DD` in America/Los_Angeles (defaults to "today" in PT)
  - `days` (defaults 30, clamp 1..30)

Response contract (DTO):
- `timezone: "America/Los_Angeles"` (constant; documents CAL-03/D-04)
- `range: { fromDate: "YYYY-MM-DD", days: number }`
- `days: Array<{ date: "YYYY-MM-DD", slots: SlotDto[] }>` where each `SlotDto` includes:
  - `slotId: string` (server-issued, stable for booking creation per D-07)
  - `startsAtUtc: string` (ISO date string in UTC)
  - `endsAtUtc: string` (ISO date string in UTC)
  - `status: "available" | "unavailable"` (Phase 3 returns only available by default; status kept for future expansion)

Implementation rules:
- Availability browsing must NOT require auth (D-06 allows auth at booking time).
- Enforce 30-day horizon in the service regardless of client input (D-02).
- Generate or ensure Slot rows exist for each day in the requested range for the given fighterId+serviceId:
  - Use `FighterScheduleRule` windows (in PT) and the selected service `durationMinutes` to compute candidate starts.
  - Use a step of 30 minutes between candidate start times.
  - Convert each candidate local PT datetime to UTC ISO string for storage in `startsAtUtc`/`endsAtUtc`.
  - Upsert into `Slot` using the unique ([fighterId, serviceId, startsAtUtc]) constraint.
- Filter to return only slots that are currently available:
  - `confirmedBookingId` is null
  - `reservedUntilUtc` is null OR `reservedUntilUtc < now`

Create `backend/src/availability/availability.constants.ts` with `AVAILABILITY_TIMEZONE = 'America/Los_Angeles'` and `MAX_DAYS = 30`.

Wire module into `backend/src/app.module.ts`.
