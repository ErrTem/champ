  <action>
Create a new Nest `BookingsModule` with a protected endpoint:

`POST /bookings`

Input DTO (`CreateBookingDto`):
- `slotId: string` (required)

Auth:
- Require session/auth (cookie refresh strategy already exists). Use the existing auth guard pattern so unauthenticated requests get a 401.

Business logic (must be concurrency-safe):
- Look up Slot by `slotId` including fighterId/serviceId and timestamps.
- Validate Slot exists; otherwise 404.
- Determine `nowUtc = new Date()` and `ttlMinutes = 15` (constant in `bookings.constants.ts`).
- In a single Prisma `$transaction`:
  1) Create Booking row with:
     - userId from auth context
     - fighterId/serviceId from Slot
     - slotId
     - status = `awaiting_payment`
     - expiresAtUtc = nowUtc + ttlMinutes
  2) Attempt to reserve the Slot atomically with a conditional update:
     - Update Slot where:
       - id = slotId
       - confirmedBookingId is null
       - reservedUntilUtc is null OR reservedUntilUtc < nowUtc
     - Set:
       - reservedUntilUtc = booking.expiresAtUtc
       - reservedBookingId = booking.id
     - If update affected 0 rows: treat as conflict (slot already reserved/confirmed). Throw an HTTP 409 with a clear error code string `SLOT_UNAVAILABLE`.
  3) If conflict thrown, ensure Booking creation is rolled back (transaction).

Response DTO (`BookingDto`):
- `id`
- `status` (must return `awaiting_payment` immediately; BKG-03)
- `expiresAtUtc` ISO string
- `slot: { slotId, startsAtUtc, endsAtUtc, fighterId, serviceId }` (enough for the UI "Booking created" screen)

Error mapping:
- Conflict/stale slot: HTTP 409 with body containing `code: "SLOT_UNAVAILABLE"` (UI uses this to show D-09 friendly error)
- Not authenticated: 401 (UI routes to login per D-06)
