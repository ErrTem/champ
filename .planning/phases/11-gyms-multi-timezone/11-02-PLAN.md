Availability timezone resolution:
- Remove or stop using global `AVAILABILITY_TIMEZONE` constant.
- Resolve `gymTimezone` per request via fighterId:
  - load fighter with included gym (per Plan 01 schema), read `fighter.gym.timezone`.
  - treat that as source of truth for all Luxon `.setZone()` and `.fromISO(..., { zone })` operations.

Parsing + generation policy (per research patterns):
- `fromDate` stays `YYYY-MM-DD` input.
- Parse: `DateTime.fromISO(fromDate, { zone: gymTimezone }).startOf('day')`
- For each local day window, interpret `FighterScheduleRule.startMinute/endMinute` as minutes since local midnight in gym timezone.
- When producing slot instants for DB, convert local DateTime to UTC via `.toUTC()` and store/query as instants.

DTO contract changes (explicit gym + timezone):
- Availability response MUST include `timezone: gymTimezone` explicitly.
- `days[].date` (or equivalent) MUST be gym-local ISO date string `YYYY-MM-DD`, computed via `DateTime.fromJSDate(startsAtUtc, { zone: 'utc' }).setZone(gymTimezone).toISODate()`.
- Slot fields continue to return UTC instants (ISO strings) as today, unless existing DTO already differs; do not introduce ambiguous local timestamps.

Safety:
- If fighter has missing gym/timezone (should not happen after Plan 01), fail fast with clear 500 and log, not silent fallback to LA.
