<threat_model>
## Trust Boundaries
| Boundary | Description |
|----------|-------------|
| Client UI → fighter self-serve API | Must not trust client role/ownership claims |
| Approved fighter → other fighters data | Must be blocked by server-side ownership checks |
| Cancellation action → payments/refunds | Must avoid false refund claims; must be consistent with Stripe handling |

## STRIDE Threat Register
| Threat ID | Category | Component | Disposition | Mitigation Plan |
|----------|----------|-----------|-------------|-----------------|
| T-12-01 | E | Self-serve endpoints | mitigate | Guard: `JwtAccessAuthGuard` + `FighterApprovedGuard`; derive fighterId server-side from `req.user.sub`. |
| T-12-02 | I | Service/schedule leakage | mitigate | Query by derived `fighterId`; verify ownership on `serviceId` updates. |
| T-12-03 | T | Cancelling wrong booking | mitigate | Cancel route checks `booking.fighterId === currentFighterId`; transactional status update + slot freeing. |
| T-12-04 | R | Refund confusion | mitigate | Implement clear policy; do not send notifications claiming refund unless actually issued. |
</threat_model>
