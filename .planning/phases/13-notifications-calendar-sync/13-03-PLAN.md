<task type="auto">
  <name>Task 3: Add service worker push handlers (minimal)</name>
  <files>angular.json,src/app/app.module.ts,src/ngsw-config.json,src/manifest.webmanifest,src/combined-sw.js,src/push-sw.js</files>
  <action>
Make Web Push actually work by ensuring a real service worker exists and has a concrete push handler, without breaking Angular SW caching.

Implementation strategy (concrete, non-magical):

1) Enable Angular service worker build
- Ensure `@angular/service-worker` is installed (add if missing).
- Update `angular.json` build config for production:
  - `serviceWorker: true`
  - `ngswConfigPath: "src/ngsw-config.json"`
- In `src/app/app.module.ts`, register SW in prod builds:
  - `ServiceWorkerModule.register('combined-sw.js', { enabled: environment.production })`

2) Add combined service worker file
- Create `src/combined-sw.js`:
  - `importScripts('ngsw-worker.js');`
  - `importScripts('push-sw.js');`
- Ensure both `combined-sw.js` and `push-sw.js` are included in build `assets` in `angular.json`.

3) Add push-only worker logic
- Create `src/push-sw.js` that implements:
  - `self.addEventListener('push', ...)` to parse payload JSON (contract from Plan 13-02: `{ type, bookingId, startAt, url }`)
  - derive **generic** `title/body` from `type` only (no names/PII)
  - `self.registration.showNotification(title, { body, data: { url }, tag: type + ':' + bookingId })`
  - `self.addEventListener('notificationclick', ...)`:
    - close notification
    - focus existing client if present, else `clients.openWindow(url ?? '/')`

Notes:
- Keep notification copy generic; do not include PII.
- Deep link URL best-effort. If route shape uncertain, use `/bookings` or `/` and let app navigate.
  </action>

  </acceptance_criteria>

<threat_model>
| Threat | Category | Mitigation |
|--------|----------|------------|
| Permission prompt spam / auto-deny patterns | UX/Abuse | Only prompt on explicit user action; provide clear explanation; do not reprompt if denied. |
| Push payload leaks data on lock screen | Info disclosure | Keep payload minimal (no phone/email/address); show generic titles; fetch details in-app. |
| ICS link shared or cached in logs | Info disclosure | Endpoint requires auth; frontend uses authenticated fetch; do not embed token in URL query; do not log URL with identifiers beyond bookingId. |
</threat_model>
