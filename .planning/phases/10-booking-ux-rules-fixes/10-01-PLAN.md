  - Use Nest `@Query()` to read:
    - `minPriceCents?: string`
    - `maxPriceCents?: string`
    - `discipline?: string | string[]` (repeatable)

  - Base filter always: `fighter.published = true`.
  - Apply `disciplines` constraint at fighter level via prisma `where: { disciplines: { hasSome: [...] } }` when requested.
  - Apply service constraints via prisma relational filter on `services` (published only):
    - For price: `services: { some: { published: true, priceCents: { gte/lte } } }`
    - For modality: `services: { some: { published: true, modality: { in: [...] } } }`
    - If both price+modality present: same `some` clause must satisfy both (service-level AND), so fighters only pass when a single service matches all service-level constraints.
  - Keep selection fields same as today (id/name/photoUrl/summary/disciplines and min published service price for `fromPriceCents`).
  - Keep ordering deterministic (name asc).

Backend: Booking 24h rule
- Add constant in `backend/src/bookings/bookings.constants.ts`:
  - `export const BOOKING_TOO_SOON_CODE = 'BOOKING_TOO_SOON' as const;`
- In `backend/src/bookings/bookings.service.ts` `createBooking()` after loading slot (after NotFound guard) and before creating booking:
  - Compute diff between `slot.startsAtUtc` and `nowUtc` (use `DateTime.utc()` or `DateTime.fromJSDate(slot.startsAtUtc).diffNow(...)` but ensure UTC).

    - throw `BadRequestException({ code: BOOKING_TOO_SOON_CODE, message: 'Bookings must be made at least 24 hours in advance.' })`

  - This avoids Nest default body shape drift (e.g. `statusCode` wrapper) and keeps contract stable for UI.

## Trust Boundaries

| Boundary | Description |
|---|---|
| client → API (`GET /fighters`) | untrusted query params cross into DB query construction |
| client → API (`POST /bookings`) | untrusted slotId attempts to reserve capacity + create booking |
| API → DB (Prisma) | filters become structured DB predicates |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-10-01 | T | `GET /fighters` query parsing | mitigate | coerce numbers/arrays safely; ignore invalid values; whitelist modality values; never pass raw strings into `$executeRaw*` |
| T-10-02 | D | `GET /fighters` filter abuse | accept | low-impact list endpoint; prisma queries bounded; rely on existing infra; keep `maxWorkers:1` only for tests |
| T-10-03 | T | `POST /bookings` business rule bypass | mitigate | enforce 24h rule server-side in `BookingsService.createBooking()` before booking creation |
| T-10-04 | I | error messages leaked | accept | message is intentionally user-facing and contains no sensitive data (no times, no IDs) |
</threat_model>
